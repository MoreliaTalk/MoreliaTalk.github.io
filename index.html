# Это рабочая доска проекта MoreliaTalk # После прохождения интенсива по разработке мессенджера на языке программирования Python у группы энтузиастов появилось желание двигаться дальше. Такое желание появилось не у одного человека, и в итоге люди организовали чат в телеграмме https://t.me/chatpythonists В результате обсуждения идей родился проект MoreliaTalk. Проект MoreliaTalk некоммерческий, создан как для само-обучения его участников, так и для отработки навыков работы в коллективе. По задумке MoreliaTalk это мессенджер с клиент-серверной архитектурой (не секрет что проект вдохновлён такими мессенджерами как WatsApp, Telegram). Изначально проект пишется на языке Python и JavaScript, но если ты решил изучить другой язык, то добро пожаловать! Используемые технологии: [FastAPI](https://fastapi.tiangolo.com) веб-фреймворк для сервера и [Qt5](https://www.qt.io/) кросс-платформенный фреймворк для десктоп приложений (Windows, Linux, iOS). В разработке MoreliaTalk принимают участие на добровольной основе. Все ключевые решения, после их обсуждения, принимаются простым большинством голосов. ## Ресурсы нашего проекта ## [GitHub](https://github.com/MoreliaTalk) репозиторий Так как среди нас есть разные люди и у них разные интересы в изучении технологий, был создан аккаунт организации MoreliaTalk. В котором созданы все наши репозитории. На каждый проект был создан свой репозиторий и определена основная команда разработчиков. ## Рабочие сервера MoreliaTalk ## [Cервер №1](http://77.50.90.123:8000/) * посмотреть [статистику](http://77.50.90.123:8000/status) Параметры подключения: ``` http://77.50.90.123:8000 ``` [Сервер на Google Cloud](http://35.228.157.42:8000) * посмотреть [статистику](http://35.228.157.42:8000/status) Параметры подключения: ``` http://35.228.157.42:8000 ``` Общие правила работы. ================ ## Правила для Trello ## - Все новые карточки создаём в списке * Ideas(To-Do) *. - После обсуждения карточки из списка * Ideas * перемещаем её в соответствующий список, например * In Progress *. - После выполнения задачи или её отмены помещаем карточку в список * Done *. - Если карточку создали по-ошибке архивируйте её. - Если решил присоединиться к задаче нажми кнопку * Присоединиться * в самой карточке, справа от поля * Описание *. - Если в карточке есть наименование сторонней библиотеки, фреймворка и т.п. то ссылка на неё обязательна. ## Правила для GitHub ## ### Основные ### - Весь новый код отправляем в develop-ветку, путём пулл-реквеста, либо создаём отдельный бранч из которого делаем пулл-реквест в develop-ветку. - Только куратор репозитория имеет право принимать решения и принимать пулл-ревесты в репозиторий. - Если есть вопросы по коду пишите напрямую куратору репозитория - мы используем английский язык для коммитов и для комментариев в коде ### Правила по написанию кода ### - Для соответствия кода единому стандарту [PEP8](https://pythonworld.ru/osnovy/pep-8-rukovodstvo-po-napisaniyu-koda-na-python.html) использовать линтеры **обязательно** ## ЧаВо ## ** Что ты такое? ** MoreliaTalk - простой мессенджер который разрабатывает группа энтузиастов в своё свободное время. Чат для разработчиков в Slack: [moreliatalk.slack.com](https://moreliatalk.slack.com) # Правила по написанию кода # Для того чтобы в результате получить код который легко читается и понимается, необходимо соблюдать внутренние правила команды. ## 1. Оформить код согласно guildlines ## Все правила по оформлению исходного кода описаны в следующих статьях. Это хоть и не маленькие статьи, но знать их необходимо. [PEP8](https://pythonworld.ru/osnovy/pep-8-rukovodstvo-po-napisaniyu-koda-na-python.html) Google style [часть 1](https://habr.com/ru/post/179271/) Google Style [часть 2](https://habr.com/ru/post/180509/) ## 2. Дополнительные требования ## _свои дополнительные правила_ _нейминг модулей, переменных, методов, классов, функций_ ## 3. Проверка кода перед pull-request ## Для проверки кода есть популярный пакет flake8. _добавить описание запуска утилиты проверки кода_ ```cmd make check ``` ## 4. Создание pull-request ## В репозитории два основных бранча: master и master-develop. master - основная ветка стабильного кода, на продакшен. master-develop - основная ветка нестабильного кода, для внедрения фич. не на продакшен, а на тест сервер. в оба эти бранча возможен пулл-реквест только после ревью одного юзера с правами maintain или admin этого репозитория. Перечень юзеров, кому доступен код-ревью определяет Админ, или Владелец. Каким образом сейчас осуществляется внесение изменений в код? Всё просто: 1. Создаёте бранч (например: develop-new_function), за основу берёте бранч master-develop. 2. Переключаетесь на вновь созданные бранч develop-new_function. 3. Вносите все задуманные изменения в код. 4. Создаёте пулл-реквест из своего бранча develop-new_function в master-develop. 5. Система автоматом выбирает юзера который будет проводить код-ревью, и который перенесёт ваш код в master-develop. 6. (ОПЦИОНАЛЬНО) Удаляете бранч develop-new_function если он вам не нужен, либо продолжаете дальше в нём работать. Остальные бранчи (текущие и созданные в будущем) - являются не основными, могут создаваться и удаляться в соответствии с правами доступа конкретного юзера. Все прочие Бранчи с наименованием начинающимся на master будут удаляться.
